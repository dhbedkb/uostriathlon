This include still doesn't know about "committee members" or "FAQ items" — only Content fields (eyebrow, title, subtitle, body, image, embed, badge, meta, qa, buttons, email) and, as of Phase 2, two things the editor now controls directly: - tile.content_order: a list of {type: "..."} entries controlling which Content fields render and in what order. Falls back to DEFAULT_ORDER below when absent, which matches the order this template has always rendered in — chosen deliberately so existing _data/content/*.yml files render identically without a migration. (This differs from the illustrative "Badge, Eyebrow, Title, Subtitle, Image…" order in the brief — Image has always rendered before Eyebrow/Title/Subtitle on this site, so the fallback matches that instead, to avoid changing the look of every existing page.) - tile.visibility: an object mapping each content type to always / hover / click / hidden. Falls back to the pre-Phase-2 behaviour: eyebrow/title/subtitle/image are always visible; body, qa, meta, email and buttons follow the tile/section's expand setting, same as before. Badge and the numbered index badge are NOT part of content_order — they're absolutely-positioned overlay tags, not flow content, so reordering them has no visual meaning. They always render exactly as before. A tile can only have ONE reveal mechanism (hover or click) at a time, same as pre-Phase-2 — an individual content field can ask for "hover" or "click", but what that actually determines is *which fields sit inside the expandable region*, not a second independent trigger. If nothing on a tile is hover/click, it renders as a plain always-visible tile (an
